# Radar

## Overview

Radar is the user's chief-of-staff / care layer. It is **not** a briefing generator and not a generic summary bot.

The desired user feeling is:

> "The agent swept the board, handled the safe things, prepared the risky things, protected me from noise, and will keep watching the rest. Only these few decisions need me."

Radar turns many private and operational signals into one ranked, source-backed attention surface:

```text
source adapters -> normalized evidence -> care-state synthesis -> top decisions -> approval-ready actions -> receipts + outcome learning -> next-run tuning
```

Use one unified Radar surface. Finance, travel, relationships, projects, email, messages, calendar, repos, and notes all feed the same queue unless the user explicitly asks for a narrow slice.

## Configuration

Radar reads its configuration from `{HERMES_HOME}/radar_config.yaml`. This file is generated by the `radar_init.py` onboarding wizard or can be written manually. Key settings:

- **`artifact_root`** — Where Radar writes packets, briefs, receipts, and the event ledger.
- **`sources`** — Which source adapters are enabled and their settings (brain, email, calendar, reminders, repos, finance, travel, messages, research).
- **`authority`** — What Radar is allowed to do without asking (read_only, draft_only, external_send, etc.).
- **`calendar`** — Calendar settings including autonomous event creation/deletion.
- **`schedule`** — Cron schedules for morning/evening sweeps and interrupt checker.

If no config exists, Radar runs with safe defaults (read_only + draft_only, all artifact paths under `{HERMES_HOME}/radar/`).

## North Star: Protect, Prepare, Propose, Close

Every Radar run must do four jobs:

| Job | What Radar owns | User feeling |
|---|---|---|
| **Protect** | Notice risks, deadlines, weird changes, failed payments, broken services, neglected people, stale loops, and source blind spots. | "Nothing important is silently slipping." |
| **Prepare** | Draft replies, handoff checklists, meeting prep, travel packets, admin notes, local task packets, and verification plans. | "The hard part is already teed up." |
| **Propose** | Present exact low-friction decisions and actions with stable approval phrases. | "I can move things with `approve 1, 3`." |
| **Close loops** | Track approved, executed, blocked, deferred, stale, no-op, and waiting states with receipts. | "I do not have to remember what happened." |

Completion criterion for any run: the final brief says what was already handled, what needs the user, what is prepared for approval, what is being watched quietly, and what will happen next.

## When to Use

- The user asks for "radar," "top of mind," "what needs my attention," "daily brief," "weekly review," "chief of staff," "care layer," "deep work shield," or "approval queue."
- A scheduled loop should sweep sources and deliver a concise care-first brief.
- The user wants proposed next actions such as messages, replies, research packets, admin handoffs, project checks, or local artifacts that can be approved by number.
- A conversation needs an operator-grade synthesis across notes, projects, finance/admin, email, messages, calendar, travel, repos, job apps, relationships, and research.
- A future product/UI needs structured items, actions, approvals, receipts, and metrics.

## When NOT to Use

- The user already gave a specific approved action, e.g. "send this exact message." Use the source-specific skill and verify.
- A single-source triage is sufficient: use `google-workspace`, `himalaya`, `github-operations`, `travel-booking-research`, etc.
- Live sources are inaccessible and there is not enough local context. Report source gaps instead of inventing a Radar.
- The only useful output is raw private content. Radar should summarize with source pointers, not dump emails, messages, finance records, or secrets.

## Run Modes and Interrupt Thresholds

Pick a mode before collecting sources. The mode determines horizon, interrupt threshold, presentation emphasis, and how aggressively to draft.

| Mode | Horizon | Interrupt threshold | Emphasis |
|---|---:|---|---|
| `morning` | last 24h + next 72h | P1+ decisions, risks, people loops, deadlines | broad care sweep and top decisions |
| `evening` | today + tomorrow | unresolved commitments, tomorrow prep | close loops and tee up tomorrow |
| `weekly` | last 7d + next 14d | strategic/stale/compounding issues | commander review, metrics, stale loops |
| `deep_work_shield` | now + next 8h | only P0/security/payment/travel/people deadlines | suppress noise, protect focus |
| `travel` | trip window | logistics changes, documents, timing, payments | concierge, checklists, buffers |
| `shipping` | active sprint/incident | repo/service/build/review blockers | project/system execution |
| `family_relationships` | next 7d | emotionally/socially costly misses | promises, birthdays, warm drafts |
| `finance_admin` | next 30d | payment, fraud, legal/tax, service risk | sentinel + manual handoff |
| `on_demand` | user-specified | user intent | adapt to prompt |

Mode rules:

1. `deep_work_shield` should be quiet by default. Interrupt only for urgent risk, same-day people obligations, travel/logistics failure, security/fraud, payment/service-suspension risk, or explicit user-requested topics.
2. `weekly` should include "what changed," stale items, acceptance metrics, false positives/negatives, and compounding opportunities.
3. `travel` should prioritize timing, documents, confirmation status, weather/packing deltas, rides, timezone conflicts, receipts/expenses, and manual-only payment/identity handoffs.
4. `family_relationships` should be draft-only unless the user explicitly approves sending; never impersonate the user.
5. `finance_admin` may observe and prepare, but financial/legal side effects require action-specific approval and often manual handoff.

## Standing Authority Profile

Default authority for Radar unless the user says otherwise:

| Authority | Default allowed? | Examples |
|---|---:|---|
| `read_only` | yes | inspect notes/wiki, local receipts, Gmail metadata/content where authenticated, calendar, GitHub status, service status, existing finance/admin emails, travel confirmations |
| `draft_only` | yes | local notes, reply drafts, meeting prep, admin checklists, travel checklists, task packets, proposed commands |
| `external_send` | approval required | send email/message/social post, invite people, submit forms |
| `runtime_mutation` | explicit scoped approval required | edit code/config, restart services, mutate cron/Kanban/Linear/GitHub |
| `financial_or_legal` | exact action-specific approval required; prefer handoff | payments, cards, billing, loans, subscriptions, tax/legal/identity, contracts, insurance submissions |
| `destructive` | explicit approval + rollback/readback required | deletes, cancellations, irreversible account changes |

Never type passwords, recovery codes, card/bank numbers, SSNs/tax IDs, API keys, or other secrets. Never click submit/pay/sign/cancel/transfer buttons unless the exact action and target were explicitly approved and the UI/source state was verified. When in doubt, stop and ask.

## Workflow

### 1. Establish scope, mode, and authority

Before collecting, record:

- `run_id`, `generated_at`, `run_mode`, and horizon.
- Domains included/excluded.
- `max_authority` and side-effect boundaries.
- Source plan and expected blind spots.
- Whether this is a normal digest or an interrupt candidate.

Completion criterion: the packet has `run_mode`, `horizon`, `domains`, `source_coverage` placeholders, `standing_authority_snapshot`, and `max_authority` before ranking items.

### 2. Collect source signals with adapters

Use original/live sources when accessible. Do not infer current source state from session history alone.

| Domain | Primary sources | Radar behavior |
|---|---|---|
| Notes/wiki | Configured wiki root files (`home.md`, `hot.md`, `todo.md`, etc.), search, status pages | Start here to preserve long-running context; cite paths. |
| Email | `google-workspace` or `himalaya` | Search/read relevant threads, classify, draft replies, do not send. |
| Calendar | Google Calendar/local adapters, plus the watcher poller | Surface prep windows, conflicts, deadlines, travel timing, follow-ups. Autonomous create/delete if config allows — see "Calendar event creation" section. |
| Reminders | Apple Reminders via `remindctl show overdue/week --json` (macOS), plus task state | Promote overdue/today/week commitments, people calls, admin/finance/manual handoffs, and checklist prep; never complete/edit/delete/create reminders without exact approval. |
| Messages | Telegram/session context, Signal/iMessage helpers where appropriate | Identify people/promises; draft only; do not send without approval. |
| Finance/admin | Email finance alerts, billing, Stripe, statements/bills/loans/insurance/tax/legal docs where authorized | Read-only sentinel; classify risk; prepare manual handoffs. |
| Travel/logistics | email/calendar travel items, airline/hotel/car/rides, travel notes | Confirmation status, check-in, documents, weather, buffers, timezone, receipts. |
| Projects/systems | repos, crons, CI/CD, project tracking where relevant | Preserve gates; distinguish configured/planned/approved from verified. |
| Job apps | Application artifacts, email confirmations, recruiter threads | Draft replies/follow-ups; prepare packets. |
| Research/social | X/xurl, RSS, arXiv, Readwise, research sources | Source-backed study/proposals; no posting without approval. |
| Relationships | messages/email/calendar/contact notes and known important dates | Warm drafts, waiting-on, birthdays/occasions, emotional labor; never impersonate. |

Adapter status values: `ok`, `partial`, `blocked_auth`, `disabled_by_policy`, `not_checked`, `error`.

Collection discipline:

1. Start from notes/wiki hot/status surfaces.
2. Query live inbox/calendar/message/finance sources only as needed for freshness.
3. Pull enough detail to propose actions, not enough to dump private content.
4. Redact account numbers, card numbers, claim/ticket/confirmation numbers, tax IDs, contact identifiers, and raw transaction lists.
5. Mark inaccessible sources honestly; do not silently skip important blind spots.

Completion criterion: every top item has a source pointer or is explicitly labeled as an assumption/question.

### 3. Normalize evidence into Radar Items

A Radar Item is a durable unit of attention, not an incoming message. Merge duplicates across sources.

Required fields:

```yaml
item_id: R1
kind: obligation | opportunity | risk | waiting_on | decision | follow_up | anomaly | relationship | metric | idea
domain: notes | finance | email | messages | calendar | repos | travel | research | relationship | job_apps | projects | personal | other
summary: one-line human-readable item
why_top_of_mind: why the user should care now
source_refs: [path/message-id/thread-id/url]
freshness: timestamp or relative age
urgency: 0-5
leverage: 0-5
risk: 0-5
tractability: 0-5
confidence: 0-5
compounding: 0-5
priority_score: computed number
owner: user | agent | external person/team | unknown
status: new | continuing | waiting | blocked | done | stale | watching | suppressed
next_check: timestamp or condition
```

Synthesis rules:

- Promote cross-source convergence: a P0 item + email thread + calendar deadline outranks a single isolated alert.
- Preserve `waiting`, `blocked`, `watching`, and `suppressed`; not everything deserves immediate action.
- Separate configured/planned/approved from actually run/verified.
- Treat deliberate no-ops as first-class outcomes.
- Avoid repeating stale items unless there is new evidence, a changed recommendation, or a clear next check.

Completion criterion: top items are mutually distinct, source-backed, and ranked by operator value, not recency alone.

### 4. Add care-layer ledgers

Radar must synthesize more than tasks. Add these ledgers where sources allow.

#### Promise / waiting-on ledger

Track:

- People waiting on the user.
- The user waiting on other people.
- "You said you'd..." commitments from notes/email/messages/calendar.
- Follow-ups that are socially, professionally, or emotionally costly to miss.
- Drafted nudges and warm replies.

Output examples:

```markdown
### People / promises
- **Recruiter** — you owe a reply by tomorrow; reply drafted. `approve 4`
- **Team lead** — warm check-in likely valuable; draft ready. `approve 5`
- **Vendor review** — waiting on external gate; no action, recheck Friday.
```

#### Meeting prep and aftermath

For upcoming or recent meetings, prepare:

- What this meeting is about.
- Relevant context and prior thread.
- Open decisions and suggested agenda.
- Prep note before meeting.
- Follow-up draft after meeting.

#### Calendar + Reminders event loop

Treat Calendar and Reminders as first-class Radar sources, not optional context looked up only during manual briefs.

- The quiet watchdog is `{HERMES_HOME}/scripts/radar_watch_sources.py` and should normally run with `--adapter all` / no args. It polls source coverage, calendar, and reminders, appends only new material events to `{artifact_root}/events.jsonl`, and stays silent for no-op/duplicate cron ticks.
- Calendar horizon: next 96h by default (`RADAR_CALENDAR_LOOKAHEAD_HOURS`), with an aggressive prep window for the next 36h (`RADAR_CALENDAR_PREP_WINDOW_HOURS`). Promote upcoming meetings/prep, travel/logistics items, admin/finance deadlines, and overlapping events/conflicts.
- Reminders horizon: overdue + week via `remindctl show overdue --json --no-input` and `remindctl show week --json --no-input`. Promote overdue/today reminders, people calls, finance/admin/manual handoffs, and current-week checklist items.
- Dedupe by stable source hash/event id. Repeated calendar/reminder items should not spam the event log; only changed/new source rows should append.
- Authority boundary: read Reminders (prepare notes/checklists/drafts; do **not** complete/edit/delete/create without approval). Calendar authority follows config — see "Calendar event creation" below. Do not click privacy/OAuth prompts without exact explicit approval.
- If Calendar or Reminders access breaks, record a `blocked` source event and summarize the gap; do not silently mark the source green.
- Verification after edits: run focused tests, `py_compile`, live `--status --json`, sanitized dry-run counts for `calendar` and `reminders`, artifact scanner, pending summary, and a duplicate `--append --cron-mode` silence check.

#### Calendar event creation

When Calendar write scope is authenticated and config allows, Radar can create calendar events. Whether events are created autonomously or approval-gated depends on config:

```yaml
# In radar_config.yaml:
calendar:
  autonomous_create: true   # create without asking
  autonomous_delete_own: true  # delete events tagged created_by: radar
  max_events_per_run: 5
  max_deletes_per_run: 3
```

Event types Radar should create:

| Type | When | Example |
|---|---|---|
| `prep_block` | 15-30 min before important meetings | "Prep: Team Sync — review agenda" |
| `deep_work_shield` | Recurring focus blocks on calendar-light days | "Deep Work — 2h uninterrupted" |
| `travel_buffer` | Transit time before flights/meetings | "Buffer: drive to airport" |
| `deadline_marker` | Bill due dates, project milestones, visa/passport deadlines | "Rent due" |
| `follow_up_reminder` | Conditional follow-ups | "If no reply from X, nudge" |
| `birthday_anniversary` | Important personal dates | "Birthday 🎂" |
| `admin_block` | Time to handle accumulated admin/finance tasks | "Admin block: pay bills, file docs" |

Action format for proposed calendar events (when approval-gated):

```yaml
action_id: A3
number: 3
item_id: R2
title: Create prep block for Team Sync
type: calendar_event
target: Google Calendar (primary)
side_effect_class: external_send
exact_proposed_output:
  summary: "Prep: Team Sync"
  start: "2026-07-04T10:30:00-04:00"
  end: "2026-07-04T11:00:00-04:00"
  description: "Review agenda, prep talking points"
approval_phrase: "approve 3"
requires_confirmation_before_execute: true
verification: "calendar list --start <start> --end <end> shows the new event"
rollback_or_safety: "calendar delete <event_id>"
```

**Authority matrix (config-driven):**

| Action | Autonomous (if config allows) | Conditions |
|---|---|---|
| Create prep blocks, focus shields, travel buffers, deadline markers, follow-up reminders | ✅ Yes | Log to event registry |
| Create admin blocks, birthday/anniversary reminders | ✅ Yes | Log to event registry |
| Delete events Radar created | ✅ Yes | Check registry — only delete events with `created_by: radar` |
| Delete/modify events the user or others created | ❌ No | Requires explicit approval |
| Send invites to other people (add attendees) | ❌ No | Requires explicit approval |
| Modify recurring events the user set up | ❌ No | Requires explicit approval |

**Event registry:** Every event Radar creates must include `created_by: radar` in the event description and be logged to `{artifact_root}/calendar_registry.jsonl`. This registry is how Radar knows which events are safe to delete autonomously.

**Smart calendar management during Radar runs:**
- At each morning/evening run, create needed prep blocks, buffers, and reminders without asking (if autonomous)
- Detect stale/outdated Radar-created events and clean them up (e.g., a prep block for a meeting that was cancelled)
- Report what was created/deleted in the "I already handled" section of the brief
- Cap autonomous creation at the configured `max_events_per_run` (default 5)
- Never delete more than `max_deletes_per_run` (default 3) events per run without flagging

#### Travel concierge

Track:

- Flight/hotel/car/rides status.
- Check-in windows and document requirements.
- Weather and packing deltas.
- Arrival/departure buffers and timezone conflicts.
- Family/group coordination.
- Receipts/expenses.
- Manual-only payment/identity steps.

**Default tracking rule:** Time-sensitive travel items — flight delays, missing baggage, hotel changes, delivery logistics — should be tracked on **both** Radar and Calendar by default. Create calendar events with delivery details, contact numbers, and addresses in the event description and location fields. Update existing events when new information arrives.

#### Finance/admin sentinel

Classify finance/admin as:

| Class | Examples | Radar behavior |
|---|---|---|
| `urgent_risk` | failed payment, fraud alert, service suspension, legal/tax deadline | escalate, prepare handoff, never type secrets |
| `upcoming_obligation` | bill due, renewal, insurance/tax/legal date | remind and prepare checklist |
| `routine_noop` | statement available, autopay active, confirmation says no action | suppress or watch |
| `opportunity` | refund, reimbursement, subscription savings, price change | propose review |
| `manual_only` | card/bank/tax/legal form | stop before sensitive fields; handoff |

#### Done while you were away

Record safe work already completed in the run:

- Checked sources.
- Dedupe/suppressed routine items.
- Drafted replies/checklists/notes locally.
- Prepared task packets.
- Filed local receipts.
- Verified service/repo/cron status read-only.

This section creates the care feeling. It should be near the top of the brief.

#### Changed since last Radar

If a previous packet exists, include:

- New.
- Escalated.
- Downgraded.
- Resolved.
- Still blocked.
- Newly stale.
- Source coverage changes.

Completion criterion: the packet includes `people_promises`, `meeting_prep`, `travel_concierge`, `finance_admin_sentinel`, `already_handled`, and `changes_since_last` arrays, even if empty or not checked.

### 5. Score and rank

Default weighted score:

```text
priority = urgency*1.2 + leverage*1.4 + risk*1.1 + tractability*0.8 + confidence*0.7 + compounding*1.3
```

Finance/admin upgrades:

- Upgrade payment failure, service suspension risk, fraud/security, tax/legal deadlines, insurance/claim blockers, travel payment/document issues, cashflow deadlines.
- Downgrade routine statements, marketing offers, already-autopaid bills, confirmations saying no action required.

Care-layer upgrades:

- Upgrade neglected people, explicit promises, same-day meetings, travel timing, high-emotion family/relationship events, and tasks that unblock future burden.
- Downgrade stale repeats, vague opportunities, low-evidence novelty, and items the user has repeatedly skipped unless new evidence appears.

Completion criterion: show 3-7 decisions/actions by default, plus quiet watch/no-op items. Avoid dumping 20 undifferentiated items.

### 6. Generate approval-ready next actions

Actions must be small enough to approve, execute, and verify.

Required action fields:

```yaml
action_id: A1
number: 1
item_id: R1
title: short action name
type: message_draft | email_reply | calendar_event | task | research | implementation | review | admin | finance_review | no_op_monitor | question | draft | meeting_prep | travel_handoff | promise_followup
target: recipient/system/path/url
side_effect_class: read_only | draft_only | external_send | runtime_mutation | financial_or_legal | destructive
exact_proposed_output: draft text, command, file path, checklist, or handoff instruction
approval_phrase: "approve 1"
requires_confirmation_before_execute: boolean
verification: how success will be checked
rollback_or_safety: how to stop/reverse/limit damage
metric_hook: outcome event to record
status: proposed
```

Action rules:

- Messages/email/social posts require exact recipient/target and exact draft. Do not send without approval.
- Runtime/code/cron/Kanban actions need explicit scoped approval and verifier commands.
- Finance/legal actions must be labeled `financial_or_legal`, state exact risk, and usually be manual handoff/checklist only.
- Questions should be decision-shaped, not generic clarification dumps.
- Stable numbers matter: `approve 2` means exactly `A2` in the latest packet, not a theme or vibe.

Completion criterion: the user can reply with `approve 1, 3`, `skip 2`, or `defer 4 until Friday` and the agent can execute without re-deriving intent.

### 7. Present a care-first brief

Lead with care, not an ops report.

Default structure:

```markdown
## Radar — Today

**Verdict:** You are mostly clear. One admin risk, two people loops, and one high-leverage project action need attention.

### I already handled
- Checked calendar, inbox, notes, project loops, repos, and finance/admin signals.
- Suppressed 11 routine/no-action items.
- Prepared 3 drafts and 1 admin checklist. Nothing sent.

### Needs you
| # | Decision | Why | Reply |
|---|---|---|---|
| 1 | Approve billing handoff checklist | possible service risk | `approve 1` |
| 2 | Send warm reply to X | they may be waiting | `approve 2` |

### Ready drafts / handoffs
1. **To X** — exact draft...
2. **Admin checklist** — exact steps, no secrets handled by agent...

### People / promises
- ...

### Admin / finance / travel
- ...

### Projects / systems
- ...

### Watching quietly
- Routine statements: no action.
- Vendor review: still waiting, recheck Friday.

### Coverage / blind spots
✅ Notes · ✅ Calendar · ✅ Email · 🟡 Finance partial · ⚪ Messages not checked

### Learning
- Last run: you skipped routine statement items, so I suppressed similar no-ops today.
```

Tone guide:

- Calm, concise, and confident.
- Use "I checked," "I prepared," "I am watching," "I need you only for..."
- Do not overstate. Say "partial" or "not checked" when true.
- Do not expose raw private content.

Completion criterion: the brief is actionable in one glance and includes exact approval grammar.

### 8. Execute approved actions

When the user replies `approve 1, 2`, parse against the latest packet. If the conversation has been compacted and Hermes appends a preserved task list, treat that list as the live execution queue: update `todo` statuses first, execute only the approved scope, and do not mark the overall follow-up complete until receipts/packet updates are actually written and read back.

Accepted forms:

```text
approve 1, 2, 3
approve A1 A3
approve 1-3
approve 2 with edit: <change>
skip 4
defer 5 until Friday
```

Execution rules:

1. Load source-specific skills for approved actions.
2. Verify prerequisites immediately before action: recipient, thread, account/service target, file path, live form state.
3. Execute only the exact approved payload/scope.
4. Read back success from the original system.
5. Write an execution receipt.
6. Report successes, partials, blockers, and next state.

For finance/legal approvals, do not interpret "approve" as permission to type secrets or submit payment/legal forms unless the exact action explicitly says so and it is safe. Prefer manual handoffs.

Completion criterion: every approved action becomes `executed_verified`, `executed_unverified`, `blocked`, `cancelled`, `partial`, `deferred`, or `failed_safe` with evidence.

### 9. Record metrics and learn

Radar improves from reviewed outcomes, not self-congratulation.

Minimum metrics:

| Metric | Meaning |
|---|---|
| `source_coverage` | checked/available/blocked sources by adapter |
| `top_item_acceptance` | items the user engaged with / items shown |
| `action_acceptance_rate` | approved actions / proposed actions |
| `action_completion_rate` | verified completions / approved actions |
| `approval_to_completion_latency` | approval -> verified outcome time |
| `false_positive_count` | irrelevant items/actions |
| `false_negative_count` | missed important items |
| `stale_item_count` | repeats without progress/new evidence |
| `burden_reduced_count` | items handled/drafted/suppressed without needing the user |
| `interrupt_precision` | urgent pings that were actually worth interrupting |
| `care_miss_count` | missed birthdays, replies, bills, meetings, travel issues |
| `prepared_but_unused_count` | drafts/checklists not used |
| `manual_handoff_success_count` | finance/legal/admin handoffs the user completed |
| `finance_alert_precision` | finance items later judged actionable / finance items shown |
| `value_receipt` | qualitative or measurable result |

Feedback labels: `useful`, `not_useful`, `false_positive`, `false_negative`, `missed`, `too_noisy`, `wrong_authority`, `good_draft_bad_timing`, `action_completed`, `action_blocked`, `care_miss`, `suppressed_correctly`, `manual_handoff_completed`.

Completion criterion: future runs can say what got better, worse, or still lacks feedback.

## Productization Principles

Design Radar as if it will become a multi-tenant assistant product.

- **One Radar surface:** narrow views are filters, not separate products.
- **Adapter boundary:** sources declare scopes, freshness, permissions, privacy tier, and failure modes.
- **Artifact contract:** JSON is stable enough for UI cards, approval buttons, replay, receipts, and evaluation.
- **Human authority ledger:** approvals are typed events with exact resolved action IDs.
- **Care UX:** lead with handled/protected/prepared; not source dump.
- **Privacy by construction:** raw private content stays in source systems or local private artifacts.
- **Finance/legal gating:** money, identity, legal/tax, contracts, and secrets are never silently automated.
- **No-op discipline:** watching/suppressing/no-op is part of care, not a missing feature.
- **Outcome learning:** tune from approved/rejected/completed/missed labels.
- **Auditability:** source refs, exact proposed payloads, approvals, execution evidence, and next checks are preserved.
- **Portability:** keep user-specific paths/preferences in local config, not the skill contract.

## Common Pitfalls

1. **Briefing instead of caring.** A long list of facts is not Radar. Lead with what was handled, what needs the user, and what is being watched.
2. **Splitting into mini-radars.** Finance/work/travel/relationships should feed one ranked queue unless a narrow slice is requested.
3. **Over-automation.** Approval-ready does not mean permission to send, publish, mutate runtime, pay, sign, delete, or enter secrets.
4. **Source laundering.** Every important claim needs a source pointer or an explicit uncertainty label.
5. **Approval ambiguity.** Stable action IDs, targets, payloads, and verification are mandatory.
6. **Nag loops.** Repeating stale items without new evidence destroys trust. Downgrade, watch, or ask whether to archive.
7. **Finance noise.** Statements/autopay confirmations should usually be suppressed; payment failures/fraud/legal/tax/service risks should escalate.
8. **People blindness.** A true care layer tracks promises, waiting-on, relationships, birthdays, meetings, reminders, and emotionally costly misses.
9. **Calendar-only or reminder-only myopia.** Calendar shows time pressure; Reminders show chosen commitments. Merge both with notes/email/messages instead of treating either as the source of truth.
10. **Reminder side effects by accident.** Reading Apple Reminders is allowed; completing, editing, deleting, or creating reminders is a mutation and needs exact approval.
11. **Disabled-channel drift.** If a source/channel is disabled by policy, mark it blocked/disabled. Do not re-enable it silently.
12. **Metrics theater.** Counts are only useful if they improve future ranking, drafts, interruption, and action quality.
13. **Overreaching on hooks for follow-up tracking.** Prefer a profile-scoped Radar closeout hook over editing global persona/root instruction files. Keep the hook local by default and preserve all existing approval gates.
14. **Half-complete follow-up reports.** If tool limits or another interruption happens before `packet.json`/`receipts.md` are updated and read back, say exactly which approved actions are verified, which artifacts exist, and which receipt update remains pending.
15. **Claiming verification without fresh evidence.** Run a fresh focused verifier after the final edit, especially when scripts/artifacts were changed. Label the result honestly as ad-hoc verification rather than suite green.
16. **Background implementation overreach.** If the user says to implement a Radar plan "in the background," treat that as approval to start a bounded tracked worker for the already-approved slice, not as approval for broader system changes.
17. **Google API Python version.** The `google_api.py` script uses PEP 604 union syntax (`str | None`) which fails on Python < 3.10. Always invoke it with a Python 3.10+ interpreter. If a call fails with `TypeError` mentioning `|`, check the Python version first.
18. **OAuth scope loss on refresh.** Profile OAuth tokens may drop scopes on refresh. When Calendar API calls fail with 403 or empty results, check the token's scopes field before diagnosing as a script bug.

## Verification Checklist

When Radar writes or patches artifacts/scripts and no canonical project test/lint/build command exists, run a focused **ad-hoc verification** before reporting completion.

- [ ] Run mode, horizon, interrupt threshold, source plan, and max authority recorded.
- [ ] Standing authority snapshot included.
- [ ] Source coverage statuses recorded, including blocked/partial/not checked.
- [ ] Top items are source-backed or explicitly uncertain.
- [ ] Care ledgers initialized: already handled, needs you, people/promises, meeting prep, travel concierge, finance/admin sentinel, projects/systems, watching quietly, changed since last run.
- [ ] Finance/legal items are redacted, classified, and gated; no secrets or money/legal movement attempted.
- [ ] Proposed actions have stable IDs/numbers, exact target/payload, side-effect class, approval phrase, verification, safety/rollback, and metric hook.
- [ ] Brief leads with "I already handled" and "Needs you," not a source dump.
- [ ] Brief includes approval grammar such as `approve 1, 2`.
- [ ] No raw private source dumps, credentials, account/card numbers, contact identifiers, or secret-like values are exposed.
- [ ] Approved actions are executed through source-specific skills and verified by readback.
- [ ] Receipts and metrics are written or at least represented in the chat output.
- [ ] If HTML/file output was requested, `brief.html` exists and the artifact refs point to it.
